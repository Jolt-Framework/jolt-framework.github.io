0. Introduction

Jolt is a lightweight, open-source framework that makes it easy to develop, deploy and maintain JAMstack applications with serverless functions. It abstracts away the complexity of provisioning and managing infrastructure, allowing the developer to focus on the business logic of their application. Jolt is targeted towards developers who are interested in the easier development and low latency that the JAMstack model provides, but who also need a secure, scalable environment to perform computation that can’t be done on the client.

Before we dive into what JAMstack architecture is and why you might want to combine it with serverless functions, let’s take a look at the architecture that is traditionally used for building web applications.

To get started with Jolt, run: npm install -g @jolt-framework

A user needs to configure aws credentials for the CLI and then, navigate to the directory the user wishes to initialize a project.

To explore Jade, make sure you have a public GitHub repository available. Jolt has been tested for use with React and Gatsby - to get started.

When ready, run: jolt init

This will check that the user has the right AWS settings, prompt the user for initial project information that will be stored locally.

Here is a list of Jolt commands available:

Command Description
jolt init Creates a local config file that will be used at deployment time to deploy a JAMstack + serverless app
jolt dev Launches a local development server to simulate lambdas on the local machine
jolt projects List all your existing JAMstack + serverless apps
jolt update Creates a new version of an existing application and updates the associated AWS
jolt rollback Allows user to go to a previous version of an application
jolt destroy Remove an app and its associated AWS infrastructure

For notes on these commands and others, please visit the documentation on our GitHub page. The following chapters will go into detail about the JAMstack + Serverless architecture, Jolt and how we expanded on the functionality of Jolt.

1.0 Traditional Web Application Architecture

The term “web application architecture” refers to all of the software and hardware components that make up a web application as well as the interactions between those components. To provide some context for the JAMstack and serverless functions, we’ll start with a look at the way web application architectures have traditionally been built.

1.1 A Client-Server Architecture

The basic premise of web communication has remained the same since the start of the web. Using a web browser (a client), a user sends a request to a server for some content. When the server receives this request, it sends back a response with the requested content.

There are a wide variety of different architectural patterns that applications can adopt, but one of the oldest and most widely used patterns is the three-tier architecture.

1.2 Three-tier Architecture

The three-tier architecture consists of three “layers” of infrastructure: the web server (seen above), the application server, and the database.

Application servers are built to process requests that require some computation and thus can’t be handled by a web server which is optimized for delivering static content. The application server can handle dynamic processes like payment processing, inventory management, or any other business logic the application needs.

With the application server providing dynamic content and the web server serving up static content, we need somewhere to actually store the data associated with this content. This brings us to the final piece of a 3 three-tier architecture, the database.

There are many different kinds of databases, but at a high level, they all have the responsibility of storing persistent data so that it can be retrieved by the application or web servers on demand and sent to clients.

1.3 Expanding Beyond the Traditional Architecture

As applications grow in scope and popularity, additional pieces of infrastructure may be needed such as:

  • Additional web, application, or database servers
  • Load balancers to distribute traffic across the additional servers
  • Caching layers
