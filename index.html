0. Introduction

Jolt is a lightweight, open-source framework that makes it easy to develop, deploy and maintain JAMstack applications with serverless functions. It abstracts away the complexity of provisioning and managing infrastructure, allowing the developer to focus on the business logic of their application. Jolt is targeted towards developers who are interested in the easier development and low latency that the JAMstack model provides, but who also need a secure, scalable environment to perform computation that can’t be done on the client.

Before we dive into what JAMstack architecture is and why you might want to combine it with serverless functions, let’s take a look at the architecture that is traditionally used for building web applications.

To get started with Jolt, run: npm install -g @jolt-framework

A user needs to configure aws credentials for the CLI and then, navigate to the directory the user wishes to initialize a project.

To explore Jade, make sure you have a public GitHub repository available. Jolt has been tested for use with React and Gatsby - to get started.

When ready, run: jolt init

This will check that the user has the right AWS settings, prompt the user for initial project information that will be stored locally.

Here is a list of Jolt commands available:

Command Description
jolt init Creates a local config file that will be used at deployment time to deploy a JAMstack + serverless app
jolt dev Launches a local development server to simulate lambdas on the local machine
jolt projects List all your existing JAMstack + serverless apps
jolt update Creates a new version of an existing application and updates the associated AWS
jolt rollback Allows user to go to a previous version of an application
jolt destroy Remove an app and its associated AWS infrastructure

For notes on these commands and others, please visit the documentation on our GitHub page. The following chapters will go into detail about the JAMstack + Serverless architecture, Jolt and how we expanded on the functionality of Jolt.

1.1 Subsection Needs a Title

The basic premise of web communication has remained the same since the start of the web. Using a web browser (a client), a user sends a request to a server for some content. When the server receives this request, it sends back a response with the requested content.

Image of client web server request response

At first, web content consisted primarily of static text, images and a few other media types. While this model is an efficient, effective way of distributing information, these purely static websites are pretty limited in what they allow users to do. In the client-server diagram above, the server (a web server) is only built to send static content to clients, but what if some kind of logical functionality was needed?

For instance, imagine an e-commerce website that sells raincoats for dogs. When a user clicks a button to buy a new Pooch Poncho, a complex sequence of events needs to be handled.

  1. Authenticate the user
  2. Verify that their payment method is valid
  3. Confirm the purchase with their bank
  4. Create and save a record of the purchase
  5. Send purchase confirmation to the user

image of a computer monitor with a shopping cart being clicked, a request being sent to a web server and then being proxied to an app server and some computation being done there

To handle these kinds of dynamic requests, a 3 tier architecture can be used.

1.2 Three-tier Architecture

Accompanying the first tier (the web server), the second tier in this architecture is an application server. Application servers are built to process requests that require some computation and thus, can’t be handled by a web server which is optimized for delivering static content. This application server could handle all kinds of dynamic processes like payment processing, inventory management, or any other business logic the application needs.

With the application server providing dynamic content and the web server serving up static content, we need somewhere to actually store the data associated with this content. In our e-commerce example, we’ll probably want somewhere to store information about our inventory, our sales history and much more. This brings us to the final piece of our 3 tier architecture, the database.

There are many different kinds of databases, but at a high level, they all have the responsibility of storing persistent data so that it can be retrieved by the application or web servers on demand and sent to clients.

Completed 3 tier image

With this completed 3 tier architecture, we now have a back end that can store data, perform computation and serve up static and dynamic content to users.

1.3 Beyond the Three-Tier Architecture

As applications grow in scope and popularity, additional pieces of infrastructure are needed. Multiple application and web servers may become necessary to handle the increased traffic and a load balancer will be needed to distribute traffic evenly across those new servers. Additional databases may be required as the quantity of data increases and operations performed on that data become more frequent. In order to provide lower latency and reduce some of the load on the servers, a caching layer could be added.

Image of a more complex architecture featuring the infrastructure mentioned above

Augmenting an application architecture in this way gives developers an enormous amount of power and control over the functioning of the application. However, with great power comes great complexity and as the complexity of the application grows, scaling and maintaining it requires more and more of an investment of resources and time.

For use cases that can’t afford the costs associated with building and maintaining this type of complex architecture, there is another solution that provides an easy to build, maintain and scale platform for deploying web applications.
